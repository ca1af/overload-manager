# Generate PR Summary

현재 브랜치의 변경사항을 분석하여 PR 요약 마크다운 파일을 생성합니다. 기존에 이미 PR_SUMMARY.md 가 존재한다면 해당 파일의 내용을 덮어씁니다.

## 절차

### 0. 커밋 범위 확인 (사용자 확인 필수)

PR 요약 생성 전에 **반드시** 사용자에게 변경사항 범위를 확인받습니다.

1. `$ARGUMENTS`가 제공되면 해당 값을 base branch로 사용합니다. 없으면 `dev`을 사용합니다.
2. 아래 명령으로 최근 커밋 목록을 출력합니다:
   ```bash
   git log <base>..HEAD --oneline -20
   ```
3. 출력된 커밋 목록을 사용자에게 보여주고 다음과 같이 질문합니다:
   > 위 커밋 목록 중 **어디서부터** 변경사항을 정리할까요?
   > - 전체 포함이면 그대로 진행할게요.
   > - 특정 커밋부터 정리하고 싶으면 해당 커밋 해시나 번호를 알려주세요.
4. 사용자 응답에 따라 분기합니다:
    - **전체 동의** (예: "전체", "그대로", "다 포함" 등): `<base>..HEAD` 전체를 대상으로 진행합니다.
    - **특정 커밋 지정** (예: 커밋 해시, 목록 번호 등): 해당 커밋을 시작점으로 `<지정커밋>..HEAD` 범위를 사용합니다.

> ⚠️ 이 단계를 건너뛰지 마세요. 사용자 확인 없이 PR 요약을 생성하지 않습니다.

### 1. 기본 정보 수집

사용자가 확정한 커밋 범위(`<start>..<end>`)를 기준으로 아래 Git 명령을 실행합니다:

- `git log <start>..HEAD --oneline` — 커밋 히스토리
- `git diff <start>...HEAD --stat` — 변경된 파일 목록과 통계
- `git diff <start>...HEAD` — 전체 diff
- `git branch --show-current` — 현재 브랜치명

### 2. 변경 내용 분석

수집한 정보를 바탕으로 다음을 파악합니다:

- **목표**: 이 PR이 해결하는 문제 또는 달성하려는 목표
- **구현 고민**: 양쪽 다 일리 있는 선택지가 있었던 진짜 트레이드오프만 기술
- **파괴적 변경**: API 신규 생성/변경, 라이브러리 코드 변경, 핵심 비즈니스 로직의 큰 변경 등
- **변경 파일 분류**: PR 목표 기준 핵심 변경만 정렬

### 3. 마크다운 파일 생성

`/Users/poby/git/backoffice/PR_SUMMARY.md` 에 아래 템플릿으로 작성합니다.

```markdown
# PR: {브랜치명에서 유추한 제목}

## 1. 이 PR이 해결하는 문제 / 목표

{이 PR의 목적을 명확하게 서술합니다. 어떤 문제를 해결하거나 어떤 기능을 추가하는지를 중심으로 작성합니다.}

## 2. 구현 중 있었던 고민

{구현 과정에서 내린 설계 결정과 그 이유를 서술합니다. 검토했지만 채택하지 않은 대안이 있다면 함께 기록합니다.}

| 고민 사항 | 선택한 방향 | 이유 |
|-----------|------------|------|
| ... | ... | ... |

## 3. 파괴적 변경 (Breaking Changes)

{API 생성/변경, 라이브러리 코드 수정, 핵심 로직의 대규모 변경 등을 나열합니다. 해당 사항이 없으면 "없음"으로 표기합니다.}

- [ ] 새로운 API 엔드포인트
- [ ] 기존 API 시그니처 변경
- [ ] 라이브러리(libs/) 코드 변경
- [ ] DB 스키마 변경
- [ ] 설정(application.yml 등) 변경

{해당 항목에 대한 상세 설명}

## 4. 변경된 파일 (중요도순)

{PR의 목표와 직접적으로 관련된 핵심 파일만 나열합니다.}

| 파일 | 변경 내용 |
|------|----------|
| `ClassName` | 설명 |
```

### 4. 작성 지침

- **언어**: 한국어로 작성합니다. 기술 용어(클래스명, 어노테이션, 프레임워크 개념)는 영어 그대로 사용합니다.
- **어조**: 간결하고 명확한 서술체(~합니다/~됩니다)를 사용합니다.
- **구현 고민**: **양쪽 다 일리 있는 선택지**가 존재했던 트레이드오프만 기술합니다. 프로젝트 컨벤션(DDD, Clean Architecture 등)에 의해 당연한 선택은 고민으로 취급하지 않습니다. 컨벤션을 따르지 않았을 때 확실한 이익이 있는 경우에만 고민으로 인정합니다.
- **파괴적 변경**: 체크리스트 중 해당하는 항목만 체크(`[x]`)하고, 해당 없는 항목은 그대로 둡니다.
- **변경 파일 작성 규칙**:
    - **핵심 변경만** 기술합니다. 부수적 변경(핵심 변경에 따라 함께 수정된 파일)은 제외합니다.
    - **파일명만** 표기합니다. 디렉토리 경로는 포함하지 않습니다. (예: `ThemeCategory`, `DefaultThemeCategoryService`)
    - **보일러플레이트는 제외**합니다. Spring Boot + JPA 개발 시 기계적으로 작성되는 파일은 핵심 변경에 포함하지 않습니다:
        - Repository 인터페이스 및 구현체 (단, 특별한 조회 로직이나 QueryDSL 사용 시에는 포함)
        - 단순 View/Projection 클래스
        - 요청/응답 DTO
        - JPA Repository 인터페이스
    - 도메인 엔티티, 서비스 로직, 컨트롤러, QueryDSL 커스텀 쿼리 등 **실질적 로직이 담긴 파일** 위주로 기술합니다.
- **diff가 큰 경우**: 모든 파일을 다 읽기보다 커밋 메시지와 diff stat을 먼저 분석하고, 핵심 파일 위주로 상세 분석합니다.

## 완료 후

### 1. 문서 결과 안내

생성된 `PR_SUMMARY.md` 파일의 경로와 주요 내용을 요약하여 알려줍니다.

### 2. PR 생성 여부 확인 (사용자 확인 필수)

문서 생성/수정이 완료되면 사용자에게 PR 직접 생성 여부를 물어봅니다. 다음과 같이 안내합니다:

> 💡 터미널에서 `gh auth login`으로 GitHub에 로그인된 상태라면, 제가 바로 PR을 생성할 수 있습니다.
> PR을 생성할까요?

사용자 응답에 따라 분기합니다:

- **생성 동의**: 아래 절차로 PR을 생성합니다.
- **거절**: PR_SUMMARY.md 생성으로 마무리합니다.

### 3. PR 생성 (사용자가 동의한 경우)

1. `gh auth status`로 GitHub 로그인 상태를 확인합니다.
    - 로그인되어 있지 않으면 `gh auth login`을 먼저 실행하라고 안내하고 중단합니다.
2. PR_SUMMARY.md의 내용을 기반으로 PR을 생성합니다:
   ```bash
   gh pr create \
     --base <base-branch> \
     --title "PR 제목" \
     --body-file /Users/poby/git/backoffice/PR_SUMMARY.md
   ```
3. 생성된 PR URL을 사용자에게 안내합니다.
